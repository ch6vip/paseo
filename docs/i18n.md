# Internationalization (i18n)

Paseo supports two UI languages: English (the source language) and Simplified Chinese (`zh-CN`). The framework is intentionally minimal — a hand-rolled Context-based provider with type-checked translation keys, no external i18n library.

**Source:** `packages/app/src/i18n/` — three files: `languages.ts`, `resources.ts`, `index.tsx`.

---

## 1. Architecture at a glance

```
┌──────────────────────────────────────────────┐
│  Language source  languages.ts               │
│  - APP_LANGUAGES = ["system","en","zh-CN"]  │
│  - resolveAppLanguage()  (system → sniff nav)│
│  - isAppLanguage()  (type guard)             │
└──────────────────────────────────────────────┘
                  ↓
┌──────────────────────────────────────────────┐
│  Dictionary  resources.ts                    │
│  - en:  as const  (key source of truth)      │
│  - zh-CN: Record<TranslationKey, string>     │
│  - resources = { en, "zh-CN": zhCN }         │
│  - {{var}} placeholder interpolation         │
└──────────────────────────────────────────────┘
                  ↓
┌──────────────────────────────────────────────┐
│  Runtime  index.tsx                          │
│  - I18nProvider  (in _layout.tsx)            │
│  - useI18n()  →  { t, language, resolved }   │
│  - t() chain: zh-CN → en → key-as-fallback   │
└──────────────────────────────────────────────┘
```

The `I18nProvider` is mounted in `packages/app/src/app/_layout.tsx` inside `RuntimeProviders`, wrapping the whole app tree.

## 2. The contract

### 2.1 `TranslationKey` is the source of truth

`en` is declared with `as const`; its keys are the only valid translation keys. `zh-CN` is `Record<TranslationKey, string>`, which **forces** every English key to have a Chinese equivalent at compile time. This is the strongest guarantee the framework provides: you cannot ship a key in English without a Chinese translation. Add a key in `en` first, then add the matching entry in `zh-CN`.

### 2.2 Language resolution

`AppLanguage` is the persisted user choice: `"system" | "en" | "zh-CN"`. `ResolvedAppLanguage` is what the runtime actually uses: `"en" | "zh-CN"`.

```ts
resolveAppLanguage("system"); // reads navigator.language
// → "zh-CN" if it starts with "zh", else "en"
```

`system` is never persisted as a resolution — it's resolved on every render. The user's `settings.language` is `"system" | "en" | "zh-CN"`.

### 2.3 `t()` fallback chain

`translate(key, language, values)`:

1. Look up `resources[language][key]`
2. Fall back to `resources.en[key]`
3. Fall back to the key string itself

The third fallback is the safety net for forgotten keys. There is **no dev-mode warning** when a key is missing in `zh-CN` — the silent fallback to `en` makes leaks easy to miss. A `__DEV__` warning is on the roadmap (see §7).

### 2.4 Interpolation

`{{varName}}` placeholders. Anything not in `values` is left as the literal placeholder text:

```ts
t("settings.host.daemon.removeHost.confirm", { host: "My Box" });
// → "Remove My Box? This will delete its saved connections."
```

`{{ }}` is a deliberate departure from i18next's `{var}` and FormatJS's ICU `{var}`. The double-brace form is unmistakable in source.

## 3. Adding a new translation

1. **Add the key to `en`** in `resources.ts` (the `as const` block). The `as const` is critical — it widens the type to a literal union, not `string`.
2. **Add the matching entry to `zh-CN`** in the same file. `Record<TranslationKey, string>` will fail to compile if you skip this.
3. **Pick a namespace.** Convention:
   - `common.*` — buttons, shared verbs ("Add", "Cancel", "Save", "Remove", "Reset", "Done", "Edit", "Error", "Settings", "Installed", "Optional", "System default").
   - `connection.*` / `pairing.*` / `provider.*` — first-run flow + add-host flow.
   - `settings.<surface>.*` — one namespace per settings section (`settings.appearance.*`, `settings.host.daemon.*`, `settings.providers.*`, `settings.shortcuts.*`).
   - `welcome.*` — welcome screen only.
   - Anything not in the UI yet does **not** get a key.
4. **Consume** via `useI18n().t("namespace.key", values?)` or `t("namespace.key", { foo: "bar" })`.
5. **Do not** create module-level `const HEADER: SheetHeader = { title: t("...") }`. That freezes the value at module load and never re-renders on language switch. Use `useMemo<SheetHeader>(() => ({ title: t("...") }), [t])` instead.

## 4. Settings persistence

`AppSettings.language` is persisted via the standard `useSettings`/`updateSettings` pipeline (see `docs/data-model.md` and `packages/app/src/hooks/use-settings/storage.ts`). The change flow is:

```
user selects "简体中文" in settings
  → useAppSettings().updateSettings({ language: "zh-CN" })
  → writeAppSettings()  (debounced)
  → loadAppSettingsFromStorage()  (next mount, hydrates from disk)
  → I18nProvider re-renders with new language
  → all useI18n() consumers pick up the new t() output
```

`isAppLanguage(value)` guards the persisted value — any unrecognized string in storage is dropped back to `DEFAULT_APP_LANGUAGE` (= `zh-CN`). A test in `storage.test.ts` covers this.

## 5. Default language

`DEFAULT_APP_LANGUAGE = "zh-CN"`. This means a fresh install is Chinese. The `system` setting resolves to Chinese when `navigator.language` starts with `zh` and to English otherwise.

This is an opinionated choice for the fork's distribution channel. **Reverting to English-default for an international release requires changing `DEFAULT_APP_LANGUAGE` and `resolveAppLanguage`'s fallback** — and updating any doc that promises a Chinese-default experience. The change is one line in `languages.ts`.

## 6. Patterns and anti-patterns

### 6.1 Pass `t` to pure functions, don't close over it

```ts
// ✅ Good — pure, testable
function buildFailureCopy(endpoint: string, error: unknown, t: Translate) {
  const title = t("connection.failure.title", { endpoint });
  ...
}

// ❌ Avoid — t becomes a dependency of every callback that uses it
const buildFailureCopy = useCallback((endpoint, error) => {
  const title = t("connection.failure.title", { endpoint });
  ...
}, [t]);
```

The first form keeps business logic independent of React. The same `t` can be threaded into `useCallback` factories that produce these helpers.

### 6.2 Records of labels → Records of keys + helper

Module-level `Record<ThemeName, string>` is a translate-dead pattern — it freezes the English label. Convert to a Record of keys + a helper that takes `t`:

```ts
// ❌ Module-level frozen label
const THEME_LABELS: Record<ThemeName, string> = { light: "Light", ... };

// ✅ Key record + helper
const THEME_LABEL_KEYS: Record<ThemeName, TranslationKey> = {
  light: "settings.appearance.theme.light",
  ...
};
const themeLabel = (value: ThemeName, t: Translate) => t(THEME_LABEL_KEYS[value]);
```

### 6.3 Plurals

There is no `count` → plural form switch. The convention is two keys: `modelCount.one` and `modelCount.other`. Use ICU-style `{{count}}` interpolation:

```ts
t("settings.providers.modelCount.one"); // "1 model"
t("settings.providers.modelCount.other", { count: 5 }); // "5 models"
```

If you need true plural forms (Chinese has no plural; English only has one; but other languages will), switch to `Intl.PluralRules` and a key-per-form lookup. Out of scope today.

### 6.4 Don't translate code, identifiers, or URLs

Translation applies to user-facing strings. `accessibilityLabel`, `title`, `<Text>`, `placeholder`, `Alert.alert(message)`, `confirmDialog({ title, message, confirmLabel })` — yes. Component names, file names, error codes, URL paths, JSON keys — no.

### 6.5 Never hard-code "Error"/"Cancel"/"Save"

These live in `common.*`. Use `t("common.error")`, `t("common.cancel")`, `t("common.save")`. Adding a new common verb: extend `common.*`, not a feature-specific namespace.

## 7. Roadmap / known gaps

- **Dev-mode missing-key warning.** A `__DEV__`-gated `console.warn` in `translate()` for the case where `language === "zh-CN"` and the key is missing would surface leaks. Risk: noisy console; should be paired with a way to suppress per-file. Tracked but not implemented.
- **Lazy dictionary loading.** Today the whole `resources` object ships in the JS bundle. At ~227 keys this is fine; at thousands, split by namespace and import on demand.
- **Type-safe `t()` calls.** The current `t(key: TranslationKey, values?)` catches typos at the key level. We do not catch typos in `values` keys. A stricter overload `(key: "settings.X", values: { foo: ... })` → only allow `foo` would be a small win.
- **More languages.** `APP_LANGUAGES` is the only place to add one; the type guard, the resolution logic, and `resources` keys are all that's needed. No runtime registry.
- **Pluralization via `Intl.PluralRules`.** See §6.3.
- **CLI i18n.** `packages/cli/bin/paseo` is a separate surface; command output, help text, and `console.log` lines are not yet localized. Likely needs a parallel dictionary since the CLI has no React tree.

## 8. Coverage status

As of the last full sweep (see commit log), the i18n integration covers:

- All `screens/settings/` pages (settings + sub-routes: connections, agents, workspaces, providers, appearance, shortcuts, integrations, permissions, diagnostics, about)
- `add-host-method-modal`, `add-host-modal`, `add-provider-modal`, `pair-link-modal`, `welcome-screen`
- `desktop/components/integrations-section.tsx`, `desktop/components/desktop-permissions-section.tsx`

Not yet covered (in priority order, by user visibility):

1. `screens/agent/*` and `message.tsx` — the main agent interaction surface
2. `screens/new-workspace-screen.tsx`, `screens/projects-screen.tsx`, `screens/sessions-screen.tsx` — workspace flows
3. `components/terminal-*`, `components/diff-*` — large but mostly non-translatable code, with some labels
4. `components/command-center.tsx` — global search/palette
5. CLI (`packages/cli/`) — separate effort
